# AGENTS

## Источники истины

- `README.md` — единая точка входа по проекту, архитектуре, использованию и правилам разработки.
- `TODO.md` — канонический журнал задач и текущего состояния работы.
- `AGENTS.md` — набор правил для ИИ-агента

Не создавай новые файлы вроде `ARCHITECTURE.md`, `STATE.md`, `PROGRESS.md`, `NOTES.md`, если пользователь явно не попросил об этом.

## Правила работы агента

- Перед изменениями изучай `README.md` и `TODO.md` и `AGENTS.md`
- Любую новую задачу или подзадачу сначала фиксируй в `TODO.md`.
- После завершения работы обновляй статус в `TODO.md`.
- Если по задаче есть важные договорённости для будущих агентов, отражай их в `TODO.md` или в этом файле.
- В `TODO.md` пиши только те задачи, которые связаны с непосредственным редактированием файлов проекта
- Перед изменениями сначала изучать существующую структуру и текущие паттерны в кодовой базе.
- Не делать лишние рефакторинги, переименования и массовые косметические правки без отдельного запроса.
- Предпочитать небольшие, локальные и понятные изменения.
- Если есть несколько путей реализации, выбирать самый простой и поддерживаемый.
- Если нужно принять архитектурное решение с заметными последствиями, сначала коротко описать варианты и риски.
- Не додумывать отдельный функционал, команды или сценарии сверх прямого запроса пользователя. Делать только то, что пользователь явно попросил, и сначала согласовывать любые расширения объёма.
- для устновки библиотек использовать виртуальную среду и `uv`
- подробно лошшировть код.

## Ожидания по изменениям

- Сохраняй функциональный стиль кода.
- Не меняй уже существующий код без янвго запроса.
- Если нужно внести изменения всуществующий код, уточни необходимость этого действия


## Ведение TODO

`TODO.md` используется как рабочий лог.

Рекомендуемые статусы:
- `todo`
- `in_progress`
- `done`
- `blocked`

## Стиль кода
- Писать код понятно и явно, без лишней магии.
- Предпочитать короткие функции и модули с одной понятной ответственностью.
- Не вводить абстракции заранее, если они не решают реальную текущую задачу.
- Соблюдать существующий стиль проекта, даже если в изоляции возможен другой вариант.

### Документирование (docstring):
- Используй многострочные docstrings в формате reStructuredText.
   - Пример оформления:

    def add(a: int, b: int) -> int:
        """Складывает два целых числа.
        :param a: Первое слагаемое
        :param b: Второе слагаемое
        :returns: Сумма a и b.
        """
        return a + b

- Примеры использования (опционально):
- Если уместно, добавляй раздел Example: в docstring.
- Не добавляй лишнего кода (например, тесты или вызовы функций), если это не запрошено.
- Пиши подробные docstrings на русском языке
- в названии функций используй префикс `_` для функций, которые не попадают в публичный api и предназначены для внутренного использовния на уровне модуля.

## Организация файлов
- Перед созданием нового файла проверять, нет ли уже подходящего места в текущей структуре.
- Не дробить код на большое количество мелких файлов без необходимости.
- Имена файлов и папок делать предсказуемыми и единообразными.
- Рядом держать связанные сущности, если это упрощает навигацию по проекту.

## Git
- Перед `git add`, `git commit` и любыми действиями по подготовке изменений проверять состав файлов, попадающих в коммит.
- При формировании коммитов делай описание кратким, без эмодзи
- в описание комита указывай основные пункты изменений на русском языке. Отделяй пункты простыми '-'
